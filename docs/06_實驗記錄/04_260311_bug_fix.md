## 1. Re 多樣性根本設計錯誤

**位置：** `config_batch_gen.py` — `rho_in_list` 設計

### 錯誤

用 `rho_in_list` 來追求 Re 多樣性，這完全沒有作用。

### 原因（Re 不變性）

```
Re_lattice  = u_lb × L_char(px) / nu_lb
Re_physical = U_phys × L_phys(m) / nu_air

兩者永遠相等，因為 dx 的定義保證了這一點：
  dx             = nu_air / (velocity_scale × nu_lb)
  velocity_scale = U_phys / u_lb
```

改變 `rho_in` → `u_lb` 增大，但 `dx` 等比縮小，`L_phys = L_char × dx` 也等比縮小。  
**兩個效果完全抵消，Re 一點都沒有改變。**

`rho_in` 只改變了「每格代表幾公尺」（物理解析度），不改變流動行為。

### Re 的有效旋鈕只有兩個

| 旋鈕                      | 作用              | 限制                                         |
| ------------------------- | ----------------- | -------------------------------------------- |
| `nu_lb` ↓                 | Re 增大（最有效） | `tau = 3ν + 0.5 ≥ 0.52`，即 `nu_lb ≥ 0.0067` |
| `L_char` ↑（更大的 mask） | Re 增大           | 計算量隨 nx×ny 增加                          |
| `rho_in`                  | **無影響**        | 只改 dx（物理解析度）                        |

### Re 預覽表（固定 rho_in=1.010，u_lb=0.0816）

| nu_lb | tau   | Re @ L=60px | Re @ L=80px | Re @ L=100px | 風況               |
| ----- | ----- | ----------- | ----------- | ------------ | ------------------ |
| 0.050 | 0.650 | 98          | 131         | 163          | 層流               |
| 0.020 | 0.560 | 245         | 327         | 408          | 低湍流             |
| 0.010 | 0.530 | 490         | 653         | 816          | 中湍流             |
| 0.007 | 0.521 | 700         | 933         | 1166         | 高湍流（安全邊界） |

### 修正：改用 `nu_lb_list`

```yaml
# master_config.yaml — 正確設計
physics_control:
  rho_in: 1.010 # 固定單一值，不再是 list
  nu_lb_list: # ← Re 多樣性的唯一正確旋鈕
    - 0.050 # Re ~130   層流
    - 0.020 # Re ~327   低湍流
    - 0.010 # Re ~653   中湍流
    - 0.007 # Re ~933   高湍流
```

## 2. `config_batch_gen.py` 其他問題

### 2-A. L_char 單位錯誤：公尺被當成像素

**錯誤程式碼：**

```python
match_l = re.search(r"L(\d+)", filename_stem)
l_char  = float(match_l.group(1))   # 解析出 63，但這是公尺，不是像素
```

**影響：** Re 被低估約 8 倍

```
px/m = (HEIGHT_PX × sub_height) / (2 × margin_m)
     = (2048 × 0.8) / (2 × 100) = 8.19 px/m

63 公尺的建築 → 實際 516 px，但 L_char 被設成 63 → 誤差 8.2 倍
```

**修正：不從檔名傳遞，直接從 PNG 像素計算**

```python
def calc_l_char_from_png(png_path, invert, nx, ny):
    img = cv2.imread(png_path, cv2.IMREAD_GRAYSCALE)
    img = cv2.resize(img, (nx, ny), interpolation=cv2.INTER_NEAREST)
    solid      = (img > 127) if invert else (img < 127)
    solid      = solid.T                      # → shape [nx, ny]
    y_occupied = np.any(solid, axis=0)        # Y 軸投影（與 physics_utils 定義一致）
    return max(1, int(np.sum(y_occupied)))
```

---

### 2-B. 入口速度高估，導致步數偏少

**錯誤程式碼：**

```python
estimated_u    = math.sqrt((2.0 / 3.0) * delta_rho)   # 理想 Bernoulli 上限
steps_per_pass = int(nx_val / estimated_u)             # 高估 u → 步數偏少
```

**修正：** 加 0.6 保守修正係數，讓步數偏大（更充裕的暖機）

```python
U_STEP_FACTOR = 0.6
u_bernoulli   = math.sqrt((2.0 / 3.0) * delta_rho)
u_for_steps   = u_bernoulli * U_STEP_FACTOR
steps_per_ctu = int(l_char / u_for_steps)   # 以 L_char（非 nx）為基準
```

---

### 2-C. `steps_per_pass` 用 `nx` 而非 `L_char`

**問題：**

- `nx / u` = 流體穿越整個場域的時間（domain pass）
- 物理正確的基準 = CTU = `L_char / u`（流體穿越特徵長度的時間）
- 兩者差了 `nx / L_char` 倍（通常 4~8 倍）

**修正：** 步數計算統一改用 `L_char / u_for_steps`（見 2-B）

---

### 2-D. 沒有 Ma 數上限與 tau 下限檢查

**危險範圍：**

| rho_in | u_lb   | Ma    | 狀態    |
| ------ | ------ | ----- | ------- |
| 1.010  | 0.0816 | 0.141 | ✅ 安全 |
| 1.020  | 0.1155 | 0.200 | ❌ 跳過 |
| 1.050  | 0.1826 | 0.316 | ❌ 必崩 |

**修正：**

```python
MA_LIMIT = 0.17   # u_max ≈ 0.098 lu/step
TAU_MIN  = 0.52   # nu_min ≈ 0.0067（舊值 0.505 距極限 tau=0.5 只有 1%，太危險）

if Ma > MA_LIMIT:
    skip(...)
if tau < TAU_MIN:
    skip(...)
```

---

### 2-E. 未驗證 passes 邏輯，可能導致 HDF5 空白

若 `start_record_passes ≥ total_passes`，模擬跑完但 HDF5 完全沒有資料，且不報錯（無聲失敗）。

**修正：在啟動前防呆**

```python
if not (w_passes < s_passes < t_passes):
    print("[Error] 必須滿足 warmup < start_record < total")
    sys.exit(1)
```

## 3. `run_one_case.py` — NameError `measured_u`

### 錯誤

```python
# 第 1 行：正確計算，存進 inlet_u
inlet_u = float(np.mean(vel_np[1, 1:-1, 0]))
metadata["u_inlet_lattice_lu"] = inlet_u

# 第 2 行：用了從未定義的 measured_u → NameError！
actual_re = (measured_u * l_char) / nu          # ← 崩潰點
metadata["u_inlet_lattice_lu"] = measured_u    # ← 多餘的重複覆蓋（且變數不存在）
```

**影響：** 所有模擬成功的 case 在最後一步都崩潰，metadata 全部寫入失敗。

### 修正

```python
vel_np   = solver.vel.to_numpy()
inlet_u  = float(np.mean(vel_np[1, 1:-1, 0]))   # 統一用 inlet_u，只定義一次

l_char    = config["simulation"]["characteristic_length"]
nu        = config["simulation"]["nu"]
actual_re = (inlet_u * l_char) / nu if nu > 0 else float("inf")

# 只寫一次，不重複覆蓋
metadata["u_inlet_lattice_lu"]             = inlet_u
metadata["reynolds_number_lattice_actual"] = actual_re
```

> 為什麼取 `vel_np[1, ...]` 而非 `vel_np[0, ...]`：  
> `x=0` 是 Zou-He 邊界節點，其分佈函數每步被 `apply_bc` 直接覆寫、不參與碰撞。  
> `x=1` 才是第一個「真正碰撞過」的流體節點，速度更具代表性。

## 4. `run_one_case.py` — `get_simulation_strategy` 覆蓋 config 步數

### 問題鏈

```
config_batch_gen.py 寫入：
  boundary_condition.value = [[0.05, 0.0], ...]   ← Zou-He 壓力邊界的 dummy 值

LBM2D_MRT_LES.__init__ 讀取：
  self.u_inlet = bc_values[0] = [0.05, 0.0]        ← dummy 被誤存為真實速度

run_one_case.py 使用：
  u_inlet_mag          = norm([0.05, 0.0]) = 0.05
  phys_steps           = nx / 0.05 × 5 = 409,600   ← 由 dummy 決定的錯誤步數
  max_steps            = min(config 設定值, 409,600) ← 你的 config 被覆蓋掉
```

**影響：** 無論 config 設定多少步數，實際都跑 409,600 步。Console 顯示的 `SIMULATION STRATEGY: 5 PASSES`、`1 Pass ~ 81920 steps` 就是這個問題的輸出。

### 修正：完全移除 `get_simulation_strategy`

```python
# 移除這三行：
# u_inlet_mag          = np.linalg.norm(solver.u_inlet)
# phys_suggested_steps, _ = utils.get_simulation_strategy(solver, u_inlet_mag)
# max_steps            = int(min(cfg_max_steps, phys_suggested_steps))

# 直接用 config 的值：
max_steps = int(config["simulation"]["max_steps"])
```

**根本原因：**  
Zou-He 壓力邊界中，`boundary_condition.value` 的速度欄位是 dummy，真實入口速度由 `rho_in - rho_out` 的壓差在模擬執行時自動決定。`solver.u_inlet` 永遠是 `0.05`，不能用來估算任何步數。

## 5. 模擬時長設定不足

### 原始設定（嚴重不足）

```yaml
warmup_passes: 1 # 有效資料窗口
total_passes: 6 # = 3 CTU
start_record_passes: 3 #
```

### 問題

都市建築繞流的流場發展需要時間，1 CTU 的暖機流場連渦流結構都還沒形成：

| CTU 範圍 | 物理事件                                 |
| -------- | ---------------------------------------- |
| 0 ~ 2    | 壓力波從入口傳到出口，純粹初始衝擊       |
| 2 ~ 5    | 主要渦流結構（尾渦、街道峽谷渦）開始形成 |
| 5 ~ 10   | 渦流脫落進入週期性 / 混沌狀態            |
| 10 ~ 20  | 時均速度場開始收斂                       |
| 20+      | Reynolds stress、湍流統計量才算可靠      |

渦流脫落週期（Strouhal St ≈ 0.2）：  
`T_vortex ≈ L_char / (St × u_lb) ≈ 3 CTU`  
→ 3 CTU 的資料窗口只涵蓋約 1 個渦流週期，統計量誤差極大。

### 你的設定 vs 建議（以 CTU 為單位）

| 參數                | 原始      | 最低標準   | 建議標準   |
| ------------------- | --------- | ---------- | ---------- |
| warmup_passes       | 1         | 5          | 8          |
| start_record_passes | 3         | 15         | 20         |
| total_passes        | 6         | 30         | 50         |
| **有效資料窗口**    | **3 CTU** | **15 CTU** | **30 CTU** |

### 建議設定

```yaml
# 快速測試 / Debug（確認不崩潰，不做統計）
warmup_passes: 2
start_record_passes: 8
total_passes: 15

# 資料集生成 — 最低標準（ML 訓練可用）
warmup_passes: 5
start_record_passes: 15
total_passes: 30

# 資料集生成 — 建議標準（Reynolds stress 基本收斂）
warmup_passes: 8
start_record_passes: 20
total_passes: 50
```

> ⚠️ 注意：新版程式改以 CTU（`L_char / u_for_steps`）為步數基準。  
> 同樣的 pass 數，新版步數比舊版（`nx / u`）少 4~8 倍，**必須相應調大 pass 數**。

## 6. Re 計算公式與縮放關係總結

### 格子 ↔ 物理縮放（`physics_scaling.py` 定義）

```
velocity_scale = U_phys / u_lb
dx             = nu_air / (velocity_scale × nu_lb)   [公尺/格]
dt             = dx / velocity_scale                  [秒/步]
L_phys         = L_char(px) × dx                     [公尺]
```

### Re 不變性

```
Re = u_lb × L_char / nu_lb   =   U_phys × L_phys / nu_air
```

改 `rho_in` → Re 不變，只改 dx  
改 `nu_lb` → Re 改變，同時影響 tau 穩定性  
改 `L_char` → Re 改變，同時影響計算量

### 穩定性邊界

```
tau_min  = 0.52   →   nu_lb_min = (0.52 - 0.5) / 3 ≈ 0.0067
Ma_max   = 0.17   →   u_lb_max  = 0.17 / sqrt(3)  ≈ 0.098
                  →   Δρ_max    = (3/2) × 0.098²  ≈ 0.014
                  →   rho_in_max ≈ 1.014
```

### Re 上限估算

```
Re_max ≈ u_lb_safe × L_char / nu_lb_min
       ≈ 0.082 × L_char / 0.0067
       ≈ 12 × L_char(px)
```

| L_char | Re 上限 |
| ------ | ------- |
| 60 px  | ~730    |
| 80 px  | ~970    |
| 100 px | ~1,220  |
| 150 px | ~1,830  |
